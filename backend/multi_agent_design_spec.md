# 法律智能多 Agent 系统设计与数据规范文档

本指南详细描述了法律 AI 系统的 **Multi-Agent 架构**、**数据需求**、及**预处理流程**，为系统从传统的单 RAG 流程向专家协作系统升级提供参考方案。

---

## 一、 智能体架构设计 (Agentic Architecture)

系统建议采用 **“主控-专家 (Supervisor-Experts)”** 模式，通过专才协作提升专业深度。

### 1.1 核心角色定义

| 角色名称 | 标识符 | 核心能力描述 | 决策逻辑 |
| :--- | :--- | :--- | :--- |
| **主控协调员** | **Supervisor** | 意图识别与任务指派 | 识别用户是通过咨询还是起草需求，按需调度专家。 |
| **法规研究员** | **Researcher** | 海量数据检索与溯源 | 专注于在知识库中找到最相关的法条和案例文书。 |
| **案情分析师** | **Analyst** | 法律三段论推理 | 结合事实与 Researcher 提供的法条，执行逻辑推演。 |
| **文书代书员** | **Drafter** | 格式化法律文书生成 | 根据 Analyst 的结论，套用标准模板生成起诉状/合同。 |
| **合规审计员** | **Auditor** | 安全审计与风险提示 | 检查输出是否合规，并强制添加必要法律风险兜底说明。 |

## 二、 数据需求与格式规范

高质量的输出依赖于高质量的输入。我们需要准备四类核心数据：

### 2.1 基础知识库 (Knowledge Base)
- **内容**：国家法律、部委规章、地方法规、典型司法案例。
- **格式**：`JSONL` (UTF-8)
- **元数据 (Metadata) 必须包含**：
  - `doc_type`: `law` 或者 `case`
  - `status`: `现行有效` / `已废止` (极其关键)
  - `law_title`: 法律全称
  - `article_number`: 条文序号（建议归一化为阿拉伯数字）

### 2.2 文书模板库 (Document Templates)
- **内容**：起诉状、欠条、劳动合同、离婚协议书、答辩状等。
- **格式**：`Markdown` + `YAML Frontmatter`
- **注意**：模版中应包含占位符（如 `{{原告信息}}`），方便 **Drafter Agent** 进行填充。

### 2.3 逻辑推理示例 (Few-shot Reasoning)
- **内容**：针对复杂问题的“三段论”推理示范。
- **格式**：`JSON` 或 `Markdown`。
- **目标**：为 **Analyst Agent** 提供少样本学习，增强其法律逻辑严密性。

## 三、 数据预处理流程 (Preprocessing Pipeline)

将原始非结构化文档 (`.docx`, `.pdf`) 转化为 AI 可理解的特征数据。

### 3.1 预处理四部曲
1. **内容解析 (Extraction)**：使用 `python-docx` 或 `pdfplumber` 提取文本，移除页眉页脚及冗余水印。
2. **文本清洗 (Cleaning)**：去除非法字符，修复全角/半角符号，统一法条编号格式。
3. **语义切片 (Semantic Chunking)**：
   - **法条**：以单个条目为最小切分单位。
   - **案例**：按“基本事实”、“法院认为”、“判决结果”进行结构化切分。
4. **编号归一化 (Normalization)**：使用 `cn2an` 工具将“第一百八十二条”等汉字编号统一映射为数字索引，提升检索效率。

---

## 四、 持续优化建议

- **时效性同步**：建议建立动态监控，每周同步法改委最新的法律废止清单。
- **纠偏机制**：引入人工反馈（RLHF）来微调 **Analyst Agent** 的推理逻辑。

> [!TIP]
> **多 Agent 的核心逻辑是分而治之**。在前期准备数据时，应优先保证条文的准确性和文书模板的覆盖广度。
